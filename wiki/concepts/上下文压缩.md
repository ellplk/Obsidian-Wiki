# 上下文压缩（Context Compression）

> 在有限 Token 窗口内维持长对话的多级压缩策略

---

## 问题背景

LLM 有固定的 context window 上限（如 200K tokens）。随着对话轮数增加，消息历史不断累积，最终会超出限制。上下文压缩就是在不丢失关键信息的前提下，压缩历史消息。

## Claude Code 的多级压缩策略

在每次 `query()` 循环的 **Phase 0 预处理管线**中，按顺序执行：

| 策略 | 函数 | 作用 |
|------|------|------|
| **工具结果持久化** | `applyToolResultBudget()` | 大工具输出写磁盘，替换为文件路径 |
| **snip 压缩** | `snipCompact()` | 裁剪旧的、大的工具结果（如读取大文件的内容） |
| **微压缩** | `microcompact()` | 压缩中间轮次的工具调用结果 |
| **上下文折叠** | `contextCollapse()` | 折叠已完成的子任务，用摘要替代详细记录 |
| **全量自动压缩** | `autocompact()` | 接近上限时触发，用 AI 总结整段历史 |

## Token 预算机制

- 系统在发送 API 请求前计算当前 token 用量
- 超过预算阈值 → 触发对应级别的压缩
- `maxOutputTokensOverride`：当输出被截断时，可动态升级 output token 上限（8k → 64k）

## 响应式压缩（Reactive Compact）

当 API 返回 `413 prompt_too_long` 错误时：
1. 先尝试 `contextCollapse` 折叠（`collapse_drain_retry`）
2. 若仍失败 → 触发 `reactiveCompact`（全量 AI 摘要）
3. `hasAttemptedReactiveCompact` 标志防止重复触发死循环

## Prompt Cache 与压缩的协作

压缩后的消息前缀如果与上一轮相同，仍可命中 [[concepts/Prompt Cache|Prompt Cache]]，节省 token 费用。
